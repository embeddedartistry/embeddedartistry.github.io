1 # libmemory
2 
3 Embedded Artistry's `libmemory` is a memory management library for embedded systems. If you have a bare metal system and want to use `malloc()`, this library is for you!
4 
5 `libmemory` provides various implemetations of the `malloc()` and `free()` functions. The primary malloc implementation is a freelist allocator which can be used on a bare-metal system. Wrappers for some RTOSes are also provided (and can be added if not already). You wil also find other useful memory functions, such as `aligned_malloc()`.
6 
7 This library is meant to be coupled with a `libc` implementation (such as the [Embedded Artistry `libc`][1]). `malloc()` and `free()` are not redefined in these headers, so you can safely use this library with your platform's existing `libc`.

21 # About the Project
22 
23 This library is meant to allow developers of embedded systems to utilize the `malloc()` and `free()` functions if their platform does not currently support it. The baseline `malloc()` implementation can be used without an RTOS or any other supporting software. Only a block of memory needs to be assigned.
24 
25 Many RTOSes provide dynamic memory allocation functionality, but these functions are not typically called `malloc()` and `free()`. Wrappers can be provdied for these RTOSes to improve code portability.
26 
27 A block of memory needs to be initially assigned using the `malloc_addblock()` function. This tells the malloc implementation what memory address and size to use for the heap.
28 
29 ```
30 // Allocate 4MB to the heap starting at memory address 0xDEADBEEF
31 malloc_addblock(0xDEADBEEF, 4 * 1024 * 1024);
32 ```
33 
34 One memory has been allcated to the heap, you can use `malloc()` and `free()` as expected.
35 
36 # Project Status
37 
38 * Basic memory allocation is implemented using the free-list strategy
39 * Cross-compiling with the Premake buildsystem is currently not implemented.
40 * Only the ThreadX RTOS has currenetly been added
41  * FreeRTOS is next on the list
42 * Tests are currently in place for `malloc()`, `free()`, `aligned_malloc()`, and `aligned_free()`
43 * The ThreadX malloc implementation is not tested
44 * No test for overlapping memory blocks exists

93 ## Installation
94 
95 Currently the best method to use this project is to build it separately and copy the contents into your tree. I will improve this method to allow easier usage as a submodule.
96 
97 Copy the `include/` directory contents into your source tree.
98 
99 Build artifacts are stored in the `buildresults` folder. There is a sub-folder for the target architecture (e.g. `x86_64`). Inside of this architecture you will find folders for ach type of `malloc` allocator (freelist, threadx, etc.), and a static library will be contained there.

109 ## Usage
110 
111 A block of memory needs to be initially assigned using the `malloc_addblock()` function:
112 
113 ```
114 void malloc_addblock(void* addr, size_t size);
115 ```
116 
117 
118 This tells the malloc implementation what memory address and size to use for the heap.
119 
120 ```
121 // Allocate 4MB to the heap starting at memory address 0xDEADBEEF
122 malloc_addblock(0xDEADBEEF, 4 * 1024 * 1024);
123 ```
124 
125 `malloc()` and `free()` will fail (return `NULL`) if no memory has been allocated. Once memory has been allocated to the heap, you can use `malloc()` and `free()` as expected.
126 
127 Multiple blocks of memory can be added using `malloc_addblock()`. The memory blocks do not have to be contiguous.
128 
129 ## Aligned Malloc
130 
131 You can allocate aligned memory using `aligned_malloc()`:
132 
133 ```
134 void* aligned_malloc(size_t align, size_t size);
135 ```
136 
137 Alignment must be a power of two!
138 
139 Aligned memory can only be free'd using `aligned_free()`:
140 
141 ```
142 void aligned_free(void* ptr);
143 ```
144 
145 For more information, see `aligned_memory.h`and [the documentation][10].
