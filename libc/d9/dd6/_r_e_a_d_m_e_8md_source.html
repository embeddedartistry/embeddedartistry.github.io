1 # libc
2 Embedded Artistry's `libc` is a C standard library implementation targeted for embedded systems.
3 
4 In order to conserve precious memory resources, this library does not supply the complete C standard library functionality. Instead, a subset of functions which are useful on bare-metal embedded systems has been selected. If you have a bare metal system and want to use the standard set of C functions, this library is for you!
5 
6 Unlike many other C libraries that I've come across, this library implements unit tests and has addressed flaws in various open-source implementations of the C standard library functions.
7 
8 Note that `malloc` and `free` are not included in this library. If you need dynamic memory allocation support, you can couple this library with the [Embedded Artistry `libmemory`][1], which contains implementations of `malloc` and `free`.

22 # About the Project
23 
24 Embedded Artistry's `libc` is intended to provide a portable set of useful C standard library functions that allows quick bringup of new bare-metal embedded systems.
25 
26 Additionally, we want to ensure a high-quality `libc` implementation by ensuring that each function has unit test coverage. Many C library function implementations are untested and contain errors. We are fighting poor implementations by adding test coverage.
27 
28 In order to conserve precious memory resources, this library does not supply the complete C standard library functionality. Instead, a subset of functions which are useful on bare-metal embedded systems has been selected. This selection has primarily been driven by my own experience in bare-metal development. If you need additional features, please open a feature request!
29 
30 The functional implementations in this library have been selected for portability and quick bringup of new systems. There may be more efficient implementations for these functions, but often they are architecture specific.
31 
32 Note that `malloc` and `free` are not included in this library. Because memory allocation schemes vary greatly with embedded systems (some not even allowing dynamic memory), you will need to supply your own implementations based on your system's needs. You can also couple this library with the [Embedded Artistry `libmemory`][1], which contains implementations of `malloc` and `free`.

34 # Project Status
35 
36 This library does not supply `errno` support at this time.
37 
38 The following portions of the C library have been implemented:
39 
40 * ctype
41 * string
42 * stdlib
43  * atoX
44  * abs, labs, llabs
45  * bsearch
46  * calloc
47  * div, ldiv, lldiv
48  * heapsort, heapsort_r
49  * imaxabs, imaxdiv
50  * qsort, qsort_r
51  * rand family
52  * realloc
53  * strtoX functions (many via `gdtoa`)
54 
55 The following architectures are currently supported:
56 
57 * x86
58 * x86_64
59 
60 Up next:
61 
62 * `assert` (`assert.h`)
63 * Hooks for `exit` and `abort` (`stdlib`)
64 * ARM architecture support
65 * Cross-compilation for ARM
66 * math.h and some definitions (INFINITY, NaN)
67 * `stdio`
68 
69 The following unit tests need to be added:
70

73  * `div`, `ldiv`
74  * `realloc`
75  * `rand` family
76  * `strnstr`
77  * `memmove`
78 
79 Maybe in the future:

87 ## Requirements
88 
89 * [Doxygen][0] must be installed to generate documentation
90 * [Premake][4] is used as the buildsystem
91  * Binaries are included for Windows, Linux, and OSX
92  * If yours is not natively supported please download a binary from the website or file a GitHub issue so I can help
93 * [`git-lfs`][7] is used to store binary files
94 * `make` and `gcc` should be installed in order to compile the files

96 ## Getting the Source
97 
98 This project uses `git-lfs`, so please install it before cloning. If you cloned prior to installing `git-lfs`, simply run `git lfs pull` after installation.
99 
100 This project is [hosted on GitHub][8]. You can clone the project directly using this command:
101 
102 ```
103 git clone --recursive git@github.com:embeddedartistry/libc.git
104 ```

106 ## Building
107 
108 The library can be built by issuing the following command:
109 
110 ```
111 make
112 ```
113 
114 This will build all targets for your current architecture.
115 
116 You can clean builds using:
117 
118 ```
119 make clean
120 ```
121 
122 And you can completely eliminate the generated `Makefile`s and buildresults using:
123 
124 ```
125 make purify
126 ```
127 
128 ### Cross-compiling
129 
130 Output is currently limited to `x86_64`, but cross-compiling for ARM will be implemented in the near future.

132 ## Installation
133 
134 Currently the best method to use this project is to build it separately and copy the contents into your tree. I will improve this method to allow easier usage as a submodule.
135 
136 Copy the `include/` directory contents into your source tree.
137 
138 Build artifacts are stored in the `buildresults` folder. There is a sub-folder for the target architecture (e.g. `x86_64`). Inside of this architecture folder you will find a `libc` folder which contains the compiled static library: `libc.a`
139 
140 Copy the desired library to your project and add the library to your link step.
141 
142 Example linker flags:
143 
144 ```
145 -lc -Lpath/to/libc.a
146 ```

148 ## Testing
149 
150 The tests for this library are written with [CMocka][3]. You can run the tests by issuing the following command:
151 
152 ```
153 make test
154 ```
155 
156 By default, test results are generated for use by the CI server and are formatted in JUnit XML. The test results XML files can be found in `buildresults/testresults/`.
157 
158 If you would prefer to see the test result summary printed to `stdout` instead, simply use the following command:
159 
160 ```
161 CMOCKA_MESSAGE_OUTPUT=stdout make test
162 ```
