2 
3 Welcome! We love receiving contributions from the community, so thanks for stopping by! There are many ways to contribute, including submitting bug reports, improving documentation, submitting feature requests, reviewing new submissions, or contributing code that can be incorporated into the project.

71 ## Feature Requests
72 
73 We welcome feature requests!
74 
75 For all requests, [please file a new GitHub issue][6]. Please provide the details about the feature/function you would like to see and why you need it. Please discuss your ideas and feedback before proceeding.
76 
77 For new features or functions which have not been implemented, please label your request with the "Feature" label. If you are requesting an improvement on an existing function, please use the "Enhancement" label.

79 Small Changes can directly be crafted and submitted to the GitHub Repository as a Pull Request. See the section regarding [Pull Request Submission Guidelines](#pull-request-guidelines).

81 ## Reporting Bugs
82 
83 Before you submit your issue, please [search the issue archive][6] - maybe your question or issue has already been identified or addressed.
84 
85 If you find a bug in the source code, you can help us by [submitting an issue to our GitHub issue tracker][6]. Even better, you can submit a Pull Request with a fix!
86 
87 This project uses an [issue template](docs/ISSUE_TEMPLATE.md), which GitHub will auto-populate when you create a new issue. Please make sure to follow the template and provide the necessary information. This will help us track down the bug as quickly as possible.

89 ## Improving Documentation
90 
91 Should you have a suggestion for the documentation, you can open an issue and outline the problem or improvement you have - however, creating the doc fix yourself is much better! By creating a new issue (or commenting on a related existing one), you let others know what you're working on. If you're making a small change (typo, phrasing) don't worry about filing an issue first - just open a Pull Request.
92 
93 I currently need the most help with improving the libc in-source Doxygen documentation and commenting. If you would like to help, please select a header in the `include/` folder and add Doxygen style comments to functions with no documentation.
94 
95 For large fixes, please build and test the documentation before submitting the PR to be sure you haven't accidentally introduced any layout or formatting issues. To generate the Doxygen documentation, run:
96 
97 ```
98 make doc
99 ```
100 
101 You can find the resulting output in `buildresults/doc` folder.

113 ### Getting Started
114 
115 Install these dependencies:
116 
117 * [Doxygen](http://www.stack.nl/~dimitri/doxygen/) must be installed to generate documentation
118 * [Premake](https://github.com/premake/premake-core/wiki) is used as the buildsystem
119  * Binaries are included for Windows, Linux, and OSX
120  * If yours is not natively supported please download a binary from the website or file a GitHub issue so I can help
121 * [`git-lfs`](https://git-lfs.github.com) is used to store binary files
122 * `make` and `gcc` should be installed in order to compile the files (XCode covers these)
123 
124 You will need to fork the main repository to work on your changes. Simply navigate to our GitHub page and click the "Fork" button at the top. Once you've forked the repository, you can clone your new repository and start making edits.
125 
126 When using `git`, it is best to isolate each topic or feature into a “topic branch”. Branches are a great way to group commits related to one feature together, or to isolate different efforts when you might be working on multiple topics at the same time.
127 
128 While it takes some experience to get the right feel about how to break up commits, a topic branch should be limited in scope to a single issue. If you are working on multiple issues, please create multiple branches and submit them for review separately.
129 
130 ```
131 # Checkout the master branch - you want your new branch to come from master
132 git checkout master
133 
134 # Create a new branch named newfeature (give your branch its own simple informative name)
135 git branch newfeature
136 
137 # Switch to your new branch
138 git checkout newfeature
139 ```
140 
141 For more information on the GitHub fork and pull-request processes, [please see this helpful guide][5].

169 ### Testing
170 
171 If you add code you need to add tests! We’ve learned the hard way that code without tests is undependable. If your pull request reduces our test coverage because it lacks tests then it will be rejected.
172 
173 This project uses CMocka as the testing framework.
174 
175 Please follow these guidelines:
176 
177 * New tests should be added to the [`test/`](test/) directory tree
178  * If you are adding a new function to an existing module, please add it in that module's tree
179  * e.g. `string/`, `stdlib/`, `ctype/`
180  * If you are adding a new module or a function in a new module, please create that directory and add the file there.
181 * Each function should have its own tests in its own C file.
182 * Each module should have a C file which coordinates the tests.
183  * e.g. [`string/string_tests.c`](test/string/string_tests.c)
184 * When adding a new test, add the following to the relevant test coordination file:
185  * An `extern` function declaration: `extern int memcpy_tests(void);`
186  * A call to this test function: `overall_result |= memcpy_tests();`
187 * When adding a new module:
188  * Please copy the test coordinator style from an existing module
189  * Add an `extern` function declaration in [main.c](test/main.c): `extern int ctype_tests(void);`
190  * Add a call to the test coordinator in [main.c](test/main.c): `overall_result |= ctype_tests();`

210 ### Whitespace Cleanup
211 
212 Don’t mix code changes with whitespace cleanup! If you are fixing whitespace, include those changes separately from your code changes. If your request is unreadable due to whitespace changes, it will be rejected.
213 
214 Please submit any whitespace cleanups in a separate pull request.

216 ### Git Commit Guidelines
217 
218 The first line of the commit log must be treated as as an email subject line. It must be strictly no greater than 50 characters long. The subject must stand on its own and not only make external references such as to relevant bug numbers.
219 
220 The second line must be blank.
221 
222 The third line begins the body of the commit message (one or more paragraphs) describing the details of the commit. Paragraphs are each separated by a blank line. Manual line breaks are not required.
223 
224 The last part of the commit log should contain all "external references", such as which issues were fixed.
225 
226 For further notes about git commit messages, [please read this blog post][7].

228 ## Pull Request Process
229 
230 When you are ready to generate a pull request for preliminary review or merging into the project, you must first push your local topic branch back up to GitHub:
231 
232 ```
233 git push origin newfeature
234 ```
235 
236 Once you've committed and pushed all of your changes to GitHub:
237  * Go to the page for your fork on GitHub
238  * Select your development branch
239  * Click the pull request button
240 
241 If you need to make any adjustments to your pull request, just push the updates to your branch. Your pull request will automatically track the changes on your development branch and update.
242 
243 Please follow the [Pull Request Template](docs/PULL_REQUEST_TEMPLATE.md) for this project. GitHub will automatically populate the pull request description with this template.
244 
245 ### Review Process
246 
247 Changes must build correctly on the Jenkins CI server, be free of formatting errors, and pass tests.
248 
249 One the changes pass the CI process, I will reveiw the source code and provide feedback. I work on a variety of projects, so please expect some delay in getting back to you with a code review. I will notify you as soon as I have seen the PR and provide insight into the expected review timeline.
250 
251 ### Addressing Feedback
252 
253 Once a PR has been submitted, your changes will be reviewed and constructive feedback may be provided. Feedback isn't meant as an attack, but to help make sure the highest-quality code makes it into our project. Changes will be approved once required feedback has been addressed.
254 
255 If a maintainer asks you to "rebase" your PR, they're saying that a lot of code has changed, and that you need to update your fork so it's easier to merge.
256 
257 To update your forked repository, follow these steps:
258 
259 ```
260 # Fetch upstream master and merge with your repo's master branch
261 git fetch upstream
262 git checkout master
263 git merge upstream/master
264 
265 # If there were any new commits, rebase your development branch
266 git checkout newfeature
267 git rebase master
268 ```
269 
270 If too much code has changed for git to automatically apply your branches changes to the new `master`, you will need to manually resolve the merge conflicts yourself.
271 
272 Once your new branch has no conflicts and works correctly, you can override your old branch using this command:
273 
274 ```
275 git push -f
276 ```
277 
278 Note that this will overwrite the old branch on the server, so make sure you are happy with your changes first!
279
